El paradigma perimetral y sus fallas

  • —El modelo castle-and-moat asumía que todo tráfico interno era confiable
  • —Un atacante que traspasa el perímetro obtiene acceso irrestricto y puede moverse lateralmente sin obstáculos
  • —En microservicios, el tráfico este-oeste entre servicios fluye sin cifrar, sin autenticar
  • —Las identidades de red (IPs) en Kubernetes son efímeras y no confiables

Zero Trust Architecture

“Ninguna entidad — usuario, dispositivo, servicio o segmento de red — debe recibir confianza implícita por su ubicación.”
NIST SP 800-207, 2020
  • 01Verificar — identidad explícita en cada solicitud, independientemente de la red
  • 02Autorizar — mínimo privilegio, acotado a la sesión y al recurso
  • 03Monitorear — supervisión continua del comportamiento de toda entidad

Componentes del modelo — PEP · PDP · PA

Solicitud │ ▼ ┌──────────────┐ ¿Permitir? ┌──────────────┐ │ PEP │ ───────────────▶ │ PDP │ │ Envoy proxy │ ◀─ Allow/Deny ── │ OPA │ └──────┬───────┘ └──────────────┘ │ (si Allow) ▲ ▼ │ Servicio destino ┌───────┴──────┐ │ PA │ │ (políticas) │ └──────────────┘
PEP — Policy Enforcement Point
Proxy Envoy (Istio). Intercepta todo el tráfico e impone la decisión.
PDP — Policy Decision Point
Open Policy Agent (OPA). Evalúa políticas Rego y devuelve Allow/Deny.
PA — Policy Administrator
Gestiona el ciclo de vida de políticas: creación, distribución, revocación.

Identidad como perímetro — SPIFFE/SPIRE y mTLS

SPIFFE / SPIRE — Identidad de workload

  • —El workload se inicia y SPIRE lo atestigua (imagen, namespace, nodo)
  • —Emite un SVID — certificado X.509 de ~1 hora de vida
  • —Renovación automática. Sin credenciales estáticas, sin rotación manual.
  • —Aplicable a microservicios, jobs CI/CD y agentes de IA

mTLS — Autenticación mutua

  • —Ambas partes presentan certificados — el servidor y el cliente
  • —Cifra el tráfico este-oeste dentro del clúster
  • —Gestionado automáticamente por el service mesh (Istio)
  • —Overhead: < 2 ms con Envoy y conexiones persistentes
07 / 10

Autorización — OAuth 2.0 / OIDC y Open Policy Agent

Identidad humana

  • —OAuth 2.0: tokens JWT emitidos por el IdP; acceso limitado sin exponer credenciales
  • —OIDC: capa de identidad y SSO; claims propagados entre servicios
  • —Flujo: API Gateway → OIDC → JWT → mTLS interno

OPA — Policy as Code

  • —Políticas en Rego, externas al código del servicio
  • —Actualizables sin redesplegar servicios
  • —Integración con Envoy vía External Authorization Filter
  • —Versionables, testeables y auditables en CI/CD
08 / 10

Service Mesh, microsegmentación y secretos efímeros

Istio / Envoy — PEP universal

  • —Envoy como sidecar junto a cada pod; intercepta todo el tráfico
  • —Impone mTLS, valida SVIDs, delega a OPA
  • —Network Policies: deny all por defecto; permitir solo lo necesario

Gestión de secretos — HashiCorp Vault

  • —Pod se autentica ante Vault con su SVID (sin contraseña estática)
  • —Vault emite credenciales dinámicas con TTL corto (~15 min)
  • —Renovación automática — cero credenciales estáticas en el sistema
Pod inicia │ ├─▶ SPIRE emite SVID │ ├─▶ SVID autentica ante Vault │ └─▶ Vault emite creds. dinámicas │ ├─▶ Envoy (sidecar) │ ├─▶ mTLS con peers │ └─▶ OPA: Allow/Deny │ └─▶ NetworkPolicy: solo tráfico explícitamente permitido

Conclusiones
  • ✓La confianza implícita por ubicación de red ya no es viable en sistemas distribuidos
  • ✓La identidad — humana y de workload — es el nuevo perímetro de seguridad
  • ✓La seguridad es un atributo transversal: cada capa asume que la anterior puede estar comprometida
  • ✓La implementación es incremental: mapear → SPIFFE → mTLS → OPA → monitoreo → Vault → DevSecOps
