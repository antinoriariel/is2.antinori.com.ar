El Modelo Castle-and-Moat 🏰

"Todo tráfico interno a la red corporativa es inherentemente confiable."
— Paradigma de seguridad perimetral tradicional

La premisa rota

  • 🔓 Un atacante que traspasa el perímetro obtiene acceso irrestricto
  • ➡️ Movimiento lateral sin obstáculos hacia servicios críticos
  • 🌐 La VPN es el único control de acceso real
  • 🔍 Tráfico inter-servicio sin cifrar, sin autenticar, sin autorizar
  • 👥 Amenazas internas con acceso amplio e irrestricto
Conclusión: cuando el perímetro falla, todo falla.

¿Qué es Zero Trust Architecture?

"Ninguna entidad — usuario, dispositivo, servicio o segmento de red — debe recibir confianza implícita por su ubicación."
— NIST SP 800-207, 2020

Premisa central: Nunca confiar. Siempre verificar.

🔐 Verificar

Identidad explícita comprobada en cada solicitud, independientemente de la red.

🎯 Autorizar

Mínimo privilegio por sesión. Solo lo necesario, solo cuando es necesario.

👁️ Monitorear

Supervisión continua del comportamiento e integridad de cada entidad.

⚠️ Asumir brecha

Diseño orientado a contener el daño, no solo a prevenir la intrusión.

Identidad como Nuevo Perímetro

La identidad reemplaza a la ubicación de red como base del control de acceso.

👤 Identidad Humana

Usuarios finales y administradores.

Protocolos: OAuth 2.0 y OpenID Connect (OIDC).
Tokens JWT propagados entre servicios.

⚙️ Identidad de Workload

Microservicios, funciones serverless, jobs CI/CD, agentes de IA.

Estándar: SPIFFE / SPIRE.
Certificados X.509 de corta vida (SVID).
En Kubernetes, una dirección IP que perteneció a un servicio legítimo puede reasignarse en segundos a otro pod. Las IPs no son identidades confiables.

SPIFFE y SPIRE — Identidad de Workload

SPIFFE: Secure Production Identity Framework For Everyone · SPIRE: implementación de referencia (CNCF)

Flujo de atestación y emisión de SVID:

1
Workload/Pod se inicia en el clúster
2
SPIRE realiza atestación: verifica nombre del pod, imagen de contenedor, namespace, nodo
3
Emite un SVID (SPIFFE Verifiable Identity Document): certificado X.509 con TTL ≈ 1 hora
4
El SVID se renueva automáticamente — sin intervención manual, sin credenciales estáticas
5
El SVID habilita mTLS entre servicios y autenticación ante Vault
✅ Sin credenciales hardcodeadas  ·  ✅ Sin rotación manual  ·  ✅ Rastro auditable completo

mTLS — Autenticación Mutua entre Servicios

AspectoTLS estándarmTLS
¿Quién se autentica?Solo el servidorAmbas partes
Tráfico cifradoNorte-SurNorte-Sur + Este-Oeste
Gestión de certificadosManual/parcialAutomática (service mesh)
Overhead de latencia—< 2 ms con conexiones persistentes

Garantías de mTLS:

  • Cada solicitud entre servicios está cifrada en tránsito
  • Cada servicio verifica criptográficamente la identidad del caller
  • Los certificados son gestionados automáticamente por el service mesh
  • Nota: mTLS autentica servicios, no usuarios — OAuth2/JWT para identidades humanas

OAuth 2.0 y OpenID Connect

OAuth 2.0

Framework de autorización.
Tokens de acceso JWT emitidos por el IdP.
Acceso limitado sin exponer credenciales.

OpenID Connect (OIDC)

Capa de identidad sobre OAuth 2.0.
SSO (Single Sign-On) entre servicios.
Claims de identidad propagados por JWT.
OIDC Federation 1.0: federación multi-dominio.

Arquitectura de producción moderna:

  Usuario ──▶ [API Gateway] ──▶ OAuth2/OIDC Validation
                                       │
                              JWT propagado a servicios
                                       │
                              [Servicio A] ─── mTLS ───▶ [Servicio B]
                                       │               (SPIFFE/SPIRE)
                                       ▼
                                  [OPA] decide autorización

Open Policy Agent — OPA

Motor de políticas de código abierto. Desacopla la autorización del código de aplicación.

# Política Rego: solo lectores pueden acceder a /api/docs/
package authz.microservices

default allow = false

allow {
    input.method == "GET"
    input.user.roles[_] == "reader"
    startswith(input.path, "/api/docs/")
}

# Pregunta compuesta: ¿Servicio A puede acceder a Recurso X en nombre de Usuario Y?
allow {
    service_authorized(input.source_service)
    user_has_permission(input.user, input.resource)
    within_session_scope(input.session)
}
✅ Políticas externas al servicio — consistencia
✅ Actualizables sin redesplegar servicios
✅ Versionables y testeables como código
✅ Integración nativa con Envoy External AuthZ

Service Mesh: Istio y Envoy

Envoy como sidecar

Desplegado junto a cada pod.
Intercepta todo el tráfico entrante y saliente.
Actúa como PEP universal.

Istio como plano de control

Gestiona la configuración de todos los sidecars.
Distribuye políticas y certificados.
Integra con SPIRE para identidad de workload.

Microsegmentación

Objetivo: contener el movimiento lateral tras cualquier brecha.

Analogía: puertas cortafuegos en un edificio — si una zona es comprometida, la amenaza no puede propagarse libremente.

Kubernetes Network Policies

spec:
  podSelector: {}
  policyTypes:
    - Ingress
    - Egress
  # Sin reglas →
  # DENIEGA TODO
Deny all por defecto.
Permitir explícitamente solo lo necesario.

Combinación con mTLS


Incluso si la Network Policy permite la comunicación entre dos pods, mTLS garantiza autenticación mutua antes de intercambiar datos.

NetworkPolicy + mTLS = segmentación + autenticación

Gestión de Secretos Efímeros

Problema: secretos hardcodeados en código fuente, imágenes de contenedor o variables de entorno.

Flujo Zero Trust con HashiCorp Vault:

①
Pod inicia → recibe SVID de SPIRE automáticamente
②
SVID se usa para autenticarse ante Vault — sin contraseña estática
③
Vault emite credenciales dinámicas con TTL corto (ej: 15 min)
④
Pod accede al recurso (base de datos, API externa) con esas credenciales
⑤
Credenciales expiran y se renuevan automáticamente — sin intervención
En ningún punto existen credenciales estáticas almacenadas en el sistema.

Conclusiones

✅ La confianza implícita por ubicación de red ya no es viable en entornos distribuidos
✅ La identidad — humana y de workload — es el nuevo perímetro de seguridad
✅ La seguridad es un atributo transversal y sistémico, no una capa final agregada
✅ Cada componente asume que la capa anterior puede estar comprometida (defensa en profundidad)
✅ El camino es incremental: mapear → identificar → mTLS → OPA → monitorear
⚠️ El costo promedio de una brecha (USD 4,44M) supera ampliamente el costo de implementar ZTA correctamente.
